# ADR 001: Autenticación con JWT

## Contexto: El sistema debe permitir que los veterinarios, asistentes y administradores ingresen a la plataforma con diferentes permisos. Se necesita una forma de autenticar y autorizar usuarios de manera segura para evitar accesos no autorizados.

##Decisión: Se implementará autenticación basada en **JSON Web Tokens (JWT)**, transmitidos en el encabezado `Authorization` de cada petición. Los tokens tendrán un tiempo de expiración corto y se complementarán con refresh tokens.

## Consecuencias
  • ✅ Seguridad mejorada, ya que las credenciales no se guardan en el servidor y los tokens expiran.
  • ⚠️ Necesidad de asegurar el almacenamiento de tokens en el cliente (ej. `HttpOnly cookies`).
  • ⚠️ Se debe implementar revocación en caso de robo de credenciales.

    • # ADR 002:Arquitectura basada en microservicios

    ## Contexto A medida que crezca la clínica y se integren nuevas funcionalidades (agenda, historias clínicas, facturación, recordatorios automáticos), el sistema monolítico puede volverse difícil de escalar y desplegar.

    ## Decisión Se optará por separar la aplicación en **microservicios**: gestión de pacientes, reservas de citas, facturación. Se comunicarán entre sí vía API REST y/o mensajería asíncrona (RabbitMQ/Kafka).

    ## Consecuencias
  • ✅ Cada servicio puede escalarse independientemente (ej. aumentar instancias de “reservas” en época de alta demanda).
  • ✅ Permite que distintos equipos trabajen en paralelo sobre servicios específicos.
  • ⚠️ Mayor complejidad en despliegue (necesidad de Kubernetes o similar).
  • ⚠️ Posibles problemas de latencia en la comunicación entre servicios.

    • # ADR 003: Separación frontend/backend

    ## Contexto: El sistema debe ser fácil de mantener, probar y extender. Actualmente el frontend y el backend están mezclados, lo cual complica la evolución del producto.

    ## Decisión: Se adoptará una arquitectura **SPA (Single Page Application)** con **React** para el frontend y **Spring Boot (Java) / Node.js (JavaScript)** para el backend. Ambos se comunicarán únicamente vía API REST.

    ## Consecuencias
  • ✅ Mayor mantenibilidad: los equipos de frontend y backend trabajan de manera independiente.
  • ✅ Posibilidad de reusar la API para aplicaciones móviles en el futuro.
  • ⚠️ Requiere configurar CORS y autenticación cruzada.
  • ⚠️ Se necesita coordinar versiones de la API para evitar romper el frontend.

    • ADR 004: Base de datos relacional

    ## Contexto: A medida que la clínica veterinaria crece y añade más pacientes, el número de registros en la base de datos aumentará exponencialmente. Las operaciones de lectura y escritura pueden volverse lentas si la base de datos no está diseñada para manejar una carga creciente. La base de datos es un punto crítico para el rendimiento y la escalabilidad de todo el sistema.

    ## Decisión: Utilizar una base de datos relacional (como PostgreSQL o MySQL) para la mayoría de los datos estructurados (pacientes, dueños, citas) y, si es necesario, considerar el uso de una base de datos NoSQL (como MongoDB) para datos con esquemas más flexibles, como el historial clínico. Además, implementar una estrategia de réplicas de lectura para distribuir la carga de consultas.

    ## Consecuencias
  • ✅ Escalabilidad: La implementación de réplicas de lectura permite a la aplicación manejar un gran número de consultas de lectura de manera eficiente, lo cual es fundamental para el historial de pacientes.
  • ✅ Rendimiento: Las consultas de lectura se distribuyen entre múltiples servidores, lo que reduce la carga del servidor de base de datos principal y mejora la velocidad de respuesta.
  • ✅ Mantenibilidad: El uso de una base de datos relacional proporciona una estructura de datos clara y fácil de entender, lo cual facilita el desarrollo de consultas.
  • ⚠️ Consistencia: La replicación de datos puede generar una ligera latencia entre el servidor principal y las réplicas, lo que podría llevar a que los datos no sean completamente consistentes en tiempo real.
  • ⚠️ Complejidad: La configuración y gestión de la replicación de bases de datos aumenta la complejidad de la infraestructura.
  • ⚠️ Costo: Se requiere más infraestructura para soportar las réplicas de la base de datos.

    • ADR 005: Elección de Lenguaje y Framework

    ## Contexto: El proyecto de gestión de la veterinaria es una aplicación empresarial que necesita ser robusta, segura y escalable a largo plazo. Se espera que el equipo de desarrollo crezca, y el proyecto requerirá un mantenimiento continuo y la adición de nuevas funcionalidades. La velocidad de desarrollo inicial es importante, pero no debe comprometer la calidad, la escalabilidad o la mantenibilidad futura.

    ## Decisión: Utilizar Java como lenguaje de programación principal y Spring Boot como framework para el backend.

    ## Consecuencias
  • ✅ Mantenibilidad: La sintaxis de Java, aunque puede ser verbosa, es conocida por su claridad y tipado estático, lo que facilita la lectura y el mantenimiento del código por parte de equipos grandes. Spring Boot promueve una arquitectura modular y organizada, lo que ayuda a evitar el "código espagueti".
  • ✅ Seguridad: Java cuenta con un ecosistema maduro de herramientas y bibliotecas de seguridad. Spring Security, en particular, ofrece una solución completa para manejar la autenticación, la autorización y la protección contra vulnerabilidades comunes (como ataques CSRF o XSS)
  • ✅ Escalabilidad: Tanto Java como Spring Boot están optimizados para manejar aplicaciones a gran escala. Spring Boot permite la creación de microservicios, lo que facilita el escalado horizontal de la aplicación a medida que la base de clientes y la carga de datos aumentan..
  • ✅ Rendimiento: Java es un lenguaje compilado que, gracias a la JVM (Java Virtual Machine) y su gestión de memoria optimizada, ofrece un rendimiento robusto. Spring Boot minimiza el tiempo de configuración y despliegue, lo que acelera el ciclo de desarrollo..
  • ✅ Ecosistema y Comunidad: Java y Spring tienen una de las comunidades de desarrolladores más grandes del mundo, con una vasta cantidad de recursos, librerías y documentación disponible. Esto reduce los riesgos de desarrollo y facilita la resolución de problemas.
  • ⚠️ Curva de Aprendizaje: Aunque Spring Boot simplifica mucho el desarrollo, el ecosistema de Spring es muy grande y complejo, lo que puede suponer una curva de aprendizaje inicial para los nuevos desarrolladores sin experiencia en el framework
  • ⚠️ Consumo de Recursos: En comparación con otros frameworks más ligeros, las aplicaciones de Spring Boot pueden tener un mayor consumo de memoria y un tiempo de arranque inicial más lento, especialmente en entornos de microservicios o contenedores.
  • ⚠️ Sintaxis Verbosa: En comparación con lenguajes más modernos como Python o Kotlin, la sintaxis de Java puede ser más prolija, lo que puede implicar escribir más código para lograr la misma funcionalidad.
